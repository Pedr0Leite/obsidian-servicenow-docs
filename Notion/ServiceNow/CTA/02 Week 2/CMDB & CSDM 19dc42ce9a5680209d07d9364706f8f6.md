# CMDB & CSDM

# The CSDM is the data framework that you follow when you set up ServiceNow applications.

The **Common Service Data Model (CSDM)** is a standard and consistent set of terms and definitions that span and can be used with all ServiceNow products. The CSDM terms and definitions enable service reporting and provide prescriptive guidelines for service modeling within the ServiceNow **Configuration Management Database (CMDB)**. The CMDB is a repository that creates and maintains the logical configurations your network infrastructure needs to support a ServiceNow service.

The CSDM data model supports multiple configuration strategies. The data model includes guidelines for using base system tables and relationships. Many ServiceNow products depend on data within this data model.

**Benefits of using the CSDM**

The CSDM is a CMDB-based framework that identifies where to place data for the products that you're using. Also, the CSDM is the standard for all ServiceNow products that use the CMDB.

The CSDM is a blueprint that you can follow when you set up your ServiceNow application.

The CSDM conceptual model contains these domains:

- Foundation
- Design
- Build (in 4.0)
- Manage Technical Services
- Sell/Consume

Each domain in the CSDM conceptual model is associated with one or more products,

## **Build**

CSDM version 4.0 includes the **Build** domain, which represents the tables used to build digital products like DevOps. The Build domain also includes a new class in the CMDB called Software Development Lifecycle Component (SDLC).

The tables in the Build domain are used to reference development details. Records in these tables are direct targets of the ITSM Incident, Problem, and Change Management processes. These tables represent the logical development details of the enterprise applications to be deployed and used by the business.

## **Design**

The Design domain represents the tables used to rationalize and manage your business applications. These tables are not operational, therefore, you cannot select them for Incident Management or Change Management.

The tables in the Design domain are also used by ServiceNow Application Portfolio Management (APM), however, you are not required to use APM to use these tables.

## **Manage Technical Services**

The Manage Technical Services domain represents the portfolio of technical services in use. These services are operational, which means that you can select them for Incident Management and Change Management.

The Manage Technical Services domain represents the tables currently used by ServiceNow IT Operations Management (ITOM), such as Service Mapping and Discovery, which let you manage configuration items (CIs) and their relationships, however, you are not required to use the tables.

## **Sell/Consume**

The Sell/Consume domain represents the portfolio of business services that may sell or consume elements of the Manage Technical Services domain, as well as the products that enable you to manage workflows and report service-related data.

The Sell/Consume domain represents the tables currently used by Service Portfolio Management (SPM) and Customer Service Management (CSM), however, you are not required to use these referenced tables.

## **Foundation**

The tables in the Foundation domain contain critical referential data, and tables that contain base data referenced from or to objects in the other CSDM domains.

While these tables aren't used in Configuration Management Database (CMDB) relationships, Foundation data is required before you use ServiceNow products or add data to the CMDB.
